This document describes every folder and source file in the repository and what each one does.

## Repository Root

| File | Purpose |
|---|---|

## `src/` — ML Pipeline Source Code

The four DVC pipeline stages are implemented here.

### `src/data/ingest.py`

**Stage:** `dvc repro` stage 1

Reads `data/raw/transactions.csv` (4.5M rows from `nickmuchi/financial-classification` on HuggingFace). Performs:
- Schema validation: asserts `description` and `category` columns exist with correct dtypes
- Null dropping: removes rows with null description or category
- Category filtering: drops rows whose category is not one of the 10 known labels
- Deduplication: removes exact `(description, category)` duplicates

Writes:
- `data/ingested/transactions.csv` — cleaned dataset
- `data/ingested/baseline_stats.json` — per-category row counts used by Airflow drift detector

### `src/data/preprocess.py`

**Stage:** `dvc repro` stage 2

Reads `data/ingested/transactions.csv`. Performs:
- Whitespace tokenisation and lowercasing (no external tokeniser dependency)
- Vocabulary building: top-10K tokens by frequency, min_freq=2, `<PAD>` at index 0, `<UNK>` at index 1
- Sequence encoding: maps tokens to indices, pads short sequences with 0, truncates long sequences to `max_seq_len=50`
- Stratified 70/15/15 train/val/test split

Writes to `data/processed/`:
- `X_train.npy`, `X_val.npy`, `X_test.npy` — encoded token index arrays, shape `(N, 50)`
- `y_train.npy`, `y_val.npy`, `y_test.npy` — integer label arrays
- `vocab.pkl` — `{token: index}` dictionary
- `label_encoder.pkl` — sklearn `LabelEncoder` mapping integer → category name
- `feature_baseline.json` — per-category proportions in training set (used by `/drift` endpoint)

### `src/models/model.py`

Defines `BiLSTMClassifier`, a PyTorch `nn.Module`:
- `Embedding(vocab_size, embedding_dim)` with PAD embedding zeroed out
- `LSTM(embedding_dim, hidden_size, num_layers=2, bidirectional=True, batch_first=True, dropout=0.75)`
- Dropout → `Linear(hidden_size * 2, hidden_size)` → ReLU → Dropout → `Linear(hidden_size, num_classes)`
- Takes an integer token-index tensor `(batch, seq_len)`, returns logits `(batch, num_classes)`

### `src/models/train.py`

**Stage:** `dvc repro` stage 3

Trains or fine-tunes `BiLSTMClassifier`:
- Loads hyperparameters from `params.yaml`
- If `FINETUNE_MODEL_PATH` env var is set: loads the checkpoint at that path and trains for 1 epoch (Run 2 / `bilstm_finetune`)
- Otherwise: initialises fresh model and trains for `params.epochs` epochs (Run 1 / `bilstm_training`)
- Logs 10 parameters, per-epoch train loss, val loss, val F1 to MLflow under experiment `SpendSense`
- Auto-promotes the logged model to `Staging` in MLflow Model Registry
- Saves best checkpoint to `models/latest_model.pt`
- Pushes `training_val_f1` and `training_duration_seconds` to Prometheus Pushgateway

Excluded from pytest coverage because it requires the full processed `.npy` arrays.

### `src/models/evaluate.py`

**Stage:** `dvc repro` stage 4

Loads `models/latest_model.pt` and `data/processed/X_test.npy`, `y_test.npy`. Computes:
- Test macro F1 and accuracy
- Per-class F1 for all 10 categories
- Confusion matrix as a PNG heatmap (logged to MLflow as an artifact)

Writes `metrics/eval_metrics.json`. Exits non-zero if macro F1 < 0.70 (CI quality gate).
Pushes `test_f1_macro` and `test_accuracy` to Prometheus Pushgateway.

Excluded from pytest coverage for the same reason as `train.py`.

## `backend/` — FastAPI Inference Service

### `backend/app/main.py`

The FastAPI application. Uses `lifespan` context manager to load the model at startup. Defines 9 endpoints:

| Endpoint | Method | What it does |
|---|---|---|
| `/health` | GET | Liveness probe. Always returns `{"status": "ok"}`. |
| `/ready` | GET | Readiness probe. Returns `{"ready": true}` only after model is loaded. |
| `/predict` | POST | Accepts `{"description": str}`. Returns `predicted_category`, `confidence`, `all_scores` (dict of 10 floats). |
| `/predict/batch` | POST | Accepts `{"descriptions": [str]}`. Returns list of prediction objects + total count. |
| `/models` | GET | Lists all FINISHED MLflow runs under the SpendSense experiment with their F1 scores, timestamps, and run types. |
| `/models/switch` | POST | Accepts `{"run_id": str}`. Downloads the model artifact from MLflow and hot-swaps it into the predictor singleton. No container restart needed. |
| `/feedback` | POST | Accepts description, predicted_category, actual_category, and optional transaction_id. Appends a JSON line to `feedback/feedback.jsonl`. |
| `/drift` | GET | Reads `feedback.jsonl`, computes `actual_category` distribution, compares against `feature_baseline.json`. Returns per-category shift and a `drift_detected` flag. Requires ≥ 100 feedback samples. |
| `/metrics` | GET | Returns Prometheus text exposition format. Scraped by Prometheus every 15 seconds. |

### `backend/app/predictor.py`

Defines `SpendSensePredictor`, the model singleton:
- `load_from_mlflow(run_id)`: downloads `latest_model.pt` + `vocab.pkl` + `label_encoder.pkl` from MLflow artifact store for the given run ID; replaces the in-memory model and preprocessing objects
- Applies `torch.quantization.quantize_dynamic()` (INT8 on LSTM and Linear layers) for CPU efficiency (~4× memory reduction vs FP32)
- `predict(description)`: tokenises, encodes, runs forward pass, applies softmax, returns top category + confidence + all 10 scores
- `list_mlflow_runs()`: queries MLflow for FINISHED runs under the SpendSense experiment

### `backend/app/monitoring.py`

Defines all Prometheus metric objects using `prometheus_client`:
- `REQUEST_COUNT` — Counter, labelled by endpoint and HTTP status
- `REQUEST_LATENCY` — Histogram, labelled by endpoint
- `PREDICTION_CATEGORY` — Counter, labelled by predicted category
- `BATCH_SIZE` — Histogram for batch predict sizes
- `FEEDBACK_TOTAL` — Counter, labelled by correctness (correct/incorrect)
- `DRIFT_SCORE` — Gauge, maximum per-category shift from last `/drift` call
- `MODEL_SWITCHES` — Counter for hot-swap calls
- `MODEL_LOADED` — Gauge (1 = model loaded, 0 = not loaded)

### `backend/app/schemas.py`

Pydantic request and response models for all endpoints:
- `PredictRequest`, `PredictResponse`, `BatchPredictRequest`, `BatchPredictResponse`
- `FeedbackRequest`, `FeedbackResponse`
- `ModelSwitchRequest`, `ModelSwitchResponse`
- `DriftResponse`, `ModelListResponse`

Ensures input validation and automatic OpenAPI documentation.

### `backend/Dockerfile`

Multi-stage Docker build for the backend. Installs Python dependencies, copies `backend/`, `models/`, and `data/processed/` (read-only bind mounts in compose). Runs `uvicorn backend.app.main:app`.

## `frontend/` — Streamlit UI

### `frontend/Home.py`

The default Streamlit page (`http://localhost:8501`):
- Text input for a single transaction description
- Six example buttons that pre-fill the input (Zomato, Uber, Netflix, Apollo, BESCOM, Amazon)
- Calls `POST /predict` and displays category + confidence + bar chart of all 10 scores
- Confidence explanation: maps score ranges to "Very High / High / Moderate / Low" language
- Post-prediction feedback form: dropdown to select correct category, submit calls `POST /feedback`

### `frontend/pages/1_Batch_Predict.py`

Three-tab interface for bulk classification:
- **CSV upload tab:** accepts CSV with a `description` column (optionally `amount`), classifies all rows, shows results table + Altair donut chart, CSV download button
- **Paste tab:** text area for one description per line, classifies all, shows results table
- **HDFC Statement tab:** accepts HDFC bank statement in `.xls` format; `_clean_hdfc_narration()` strips bank-specific prefixes (`UPI/`, `NEFT/`, `RTGS/`, `IMPS/`, `POS `, `ATM `, etc.) before classifying; filters to debit/withdrawal rows only

### `frontend/pages/2_Pipeline_Status.py`

Operations dashboard:
- **Service health grid**: checks 7 services (`/health` or `/ready` endpoints) and displays ✅ / ⚠️ / ❌
- **Live metrics**: calls `GET /metrics`, parses Prometheus text format, displays key counters (requests, errors, latency, drift score)
- **DVC DAG diagram**: runs `dvc dag --dot` subprocess, renders via Graphviz
- **Airflow run history**: queries Airflow REST API for recent DAG runs; expands each run to show per-task status
- **Direct links** to all tool UIs

### `frontend/monitoring.py`

Pushgateway metrics for the Streamlit frontend:
- `UI_PREDICTIONS_TOTAL` — counts predictions made via the UI
- `UI_ERRORS_TOTAL` — counts backend connection failures
- `UI_BATCH_ITEMS_TOTAL` — counts items processed via batch predict
All three push to `http://pushgateway:9091`.

## `airflow/` — Data Ingestion DAG

### `airflow/dags/ingestion_dag.py`

Defines `spendsense_ingestion_pipeline`, a 9-task Airflow DAG with `@daily` schedule.

**Task implementations:**

| Task | Function | What it does |
|---|---|---|
| `verify_raw_data` | `task_verify_raw_data` | Checks that `data/raw/transactions.csv` and `data/raw/transactions_90.csv` exist on disk |
| `validate_schema` | `task_validate_schema` | Opens transactions.csv, asserts `description` and `category` columns are present |
| `check_nulls` | `task_check_nulls` | Counts null values per column and logs the result |
| `check_drift` | `task_check_drift` | Loads `baseline_stats.json` (from ingest stage), computes per-category distribution of `transactions_drift.csv`, flags any shift > 10 pp |
| `route_on_drift` | `task_route_on_drift` (BranchPythonOperator) | Routes to `combine_data` if drift detected, else to `pipeline_complete` |
| `combine_data` | `task_combine_data` | Merges `transactions_90.csv` + `transactions_drift.csv` + `feedback/feedback.jsonl` corrections into `data/raw/transactions.csv` |
| `run_ingest` | `task_run_ingest` | Runs `src/data/ingest.py` as subprocess; skipped (`{"skipped": True, "reason": "ci_mode"}`) when `GITHUB_ACTIONS=true` because DVC Run 2 handles ingest |
| `trigger_dvc` | `task_trigger_dvc` | Dispatches `workflow_dispatch` event to GitHub Actions to trigger retraining; skipped when `GITHUB_ACTIONS=true` |
| `pipeline_complete` | `task_pipeline_complete` | Terminal task with `trigger_rule="none_failed_min_one_success"`; pushes `pipeline_complete=1.0` to Pushgateway |

### `airflow/Dockerfile`

Builds the Airflow container: installs `apache-airflow`, project dependencies, and copies the DAG file.

### `airflow/entrypoint.sh`

Docker entrypoint: initialises the Airflow SQLite database (`airflow db init`), creates the admin user if it doesn't exist, then starts Airflow in standalone mode (`airflow standalone`).

### `airflow/requirements.txt`

Additional pip packages required inside the Airflow container (pandas, requests, prometheus_client).

## `monitoring/` — Observability Configuration

### `monitoring/prometheus.yml`

Prometheus scrape configuration:
- Scrapes FastAPI backend at `backend:8000/metrics` every 15 seconds
- Scrapes Pushgateway at `pushgateway:9091` every 15 seconds
- Scrapes Prometheus itself (self-monitoring)
- Points to `alert_rules.yml` for alerting rules

### `monitoring/alert_rules.yml`

11 Prometheus alerting rules:
- `HighErrorRate` — error rate > 5% for 2 min
- `ModelNotLoaded` — model_loaded gauge = 0 for > 1 min
- `HighPredictionLatency` — P95 > 500ms for 5 min
- `LowTestF1` — test F1 < 0.70 (fires immediately)
- `LowValF1` — validation F1 < 0.65 during training
- `TrainingDurationHigh` — training duration > 2 hours
- `DataDriftDetected` — Airflow drift flag > 0
- `IngestFailed` — ingest success metric = 0
- `FeedbackLoopDead` — no new feedback in 48h
- `TailLatencySpike` — P99 latency > 1s for 5 min
- `FrequentModelSwitch` — > 3 hot-swaps per hour

### `monitoring/alertmanager.yml`

Alertmanager routing configuration. Routes all alerts to an email receiver via Gmail SMTP. SMTP password is read from `ALERTMANAGER_SMTP_PASSWORD` environment variable (leave empty to disable email).

### `monitoring/alertmanager-entrypoint.sh`

Writes the Alertmanager config file at container startup, substituting the `ALERTMANAGER_SMTP_PASSWORD` environment variable into the YAML template.

### `monitoring/grafana/provisioning/`

Grafana auto-provisioning configuration:
- `datasources/`: Prometheus datasource pointing to `prometheus:9090`
- `dashboards/`: JSON dashboard definition with 7 panels (Request Rate, Error Rate, Feedback Count, Drift Score, Latency Percentiles, Model Info, Alert Firing History)

## `scripts/` — Utility Scripts

### `scripts/create_drift_split.py`

Creates the 90/10 stratified train/drift split used by CI and Airflow:
- Reads `data/raw/transactions.csv`
- Stratified 90% sample → `data/raw/transactions_90.csv` (baseline used in DVC Run 1)
- Remaining 10% with intentional category oversampling → `data/drift/transactions_drift.csv` (simulates distribution shift for Airflow drift detection demo)

## `.github/workflows/ci.yml` — CI/CD Pipeline

Three-job GitHub Actions workflow triggered on every push:

**Job 1 (`test`, ~40s):** Runs on every branch.
1. Checkout, set up Python 3.11, install root `requirements.txt`
2. `flake8` lint check
3. `pytest` with coverage (must meet 60% threshold)

**Job 2 (`ml-pipeline`, ~11 min):** Main branch only.
1. `scripts/create_drift_split.py` → 90/10 split
2. Start infra services (MLflow, Prometheus, Grafana, Alertmanager, Pushgateway)
3. `dvc repro` → Run 1 (`bilstm_training`)
4. Save `models/latest_model.pt` → `models/run1_model.pt`
5. Build Airflow Docker image + start Airflow container
6. Verify Prometheus received training metrics
7. Run evaluation metric check (F1 ≥ 0.70)
8. Push DVC artifacts
9. Trigger Airflow DAG via REST API (wait up to 205 seconds for completion)
10. `FINETUNE_MODEL_PATH=models/run1_model.pt dvc repro --force` → Run 2 (`bilstm_finetune`)
11. Upload artifacts: `models/latest_model.pt`, `data/processed/vocab.pkl`, `data/processed/label_encoder.pkl`, `data/processed/feature_baseline.json`, `mlruns/mlflow.db`, `params.yaml`

**Job 3 (`app`, ~1.5 min):** Main branch only, depends on Job 2.
1. Download Job 2 artifacts (~30 MB total)
2. Start MLflow, then FastAPI + Streamlit containers
3. Smoke test: poll `/ready` until model loaded, run 5 predict requests, assert all return correct category + confidence
